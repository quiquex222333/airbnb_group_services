# Common Headers Standard

## Objective

Define the shared mandatory and optional HTTP headers used across all platform services in the Airbnb clone architecture.

## Scope

This standard applies to:

- API Gateway
- Auth Service
- User Service
- Listing Service
- Search Service
- Availability Service
- Booking Service
- Payment Service
- Notification Service

## Standard Headers

| Header | Required | When it applies | Example | Purpose |
|---|---|---|---|---|
| `Authorization` | Yes | Protected endpoints | `Bearer <token>` | Authenticates the caller |
| `Content-Type` | Yes | Requests with body | `application/json` | Indicates request body format |
| `Accept` | Yes | All client requests | `application/json` | Indicates expected response format |
| `X-Request-Id` | Yes | All external requests | `01ARZ3NDEKTSV4RRFFQ69G5FAV` | Identifies the original request |
| `X-Correlation-Id` | Yes | External and internal requests | `01ARZ3NDEKTSV4RRFFQ69G5FAA` | Correlates calls across microservices |

## Header Rules

### Authorization

- Required on protected endpoints.
- Must use the format: `Bearer <token>`.
- If missing on a protected endpoint, the service must respond with `401 Unauthorized`.
- If invalid or expired, the service must respond with `401 Unauthorized`.
- If valid but insufficient for the operation, the service must respond with `403 Forbidden`.

### Content-Type

- Required for `POST`, `PUT`, and `PATCH` requests that include a request body.
- Supported value: `application/json`.
- If an unsupported media type is provided, the service must respond with `415 Unsupported Media Type`.

### Accept

- Required in client-to-platform requests.
- Supported value: `application/json`.
- If the client requests an unsupported response format, the service must respond with `406 Not Acceptable`.

### X-Request-Id

- Identifies the original incoming HTTP request.
- Must be present in every external request.
- If missing, the API Gateway must generate it.
- Must be logged by every service that handles the request.

### X-Correlation-Id

- Identifies the distributed transaction across services.
- Must be present in all external and internal service calls.
- If missing in the original client request, the API Gateway must generate it.
- Must be propagated unchanged in all service-to-service communication.
- Must be included in logs, traces, and async events where applicable.

## X-Request-Id vs X-Correlation-Id

These headers are related but serve different purposes:

- `X-Request-Id`: identifies the original HTTP request.
- `X-Correlation-Id`: links the full cross-service flow for observability.

In simple flows they may initially share the same value, but they represent different concepts and should be documented separately.

## Enforcement Rules

| Case | Expected behavior |
|---|---|
| Missing `Authorization` on protected endpoint | `401 Unauthorized` |
| Invalid or expired token | `401 Unauthorized` |
| Valid token without permission | `403 Forbidden` |
| Unsupported `Content-Type` | `415 Unsupported Media Type` |
| Unsupported `Accept` | `406 Not Acceptable` |
| Missing `X-Request-Id` | Generated by API Gateway |
| Missing `X-Correlation-Id` | Generated by API Gateway |

## Responsibilities by Layer

### API Gateway

The API Gateway must:

- validate `Accept`
- validate `Content-Type` when applicable
- validate presence and format of `Authorization` on protected routes
- generate `X-Request-Id` if missing
- generate `X-Correlation-Id` if missing
- forward both tracing headers to internal services

### Microservices

Each microservice must:

- accept and log `X-Request-Id`
- accept, log, and propagate `X-Correlation-Id`
- support JSON request/response conventions
- enforce `Authorization` where the endpoint is protected
- document required headers in its API contract

## Per-Service Usage Matrix

| Service | Authorization | Content-Type | Accept | X-Request-Id | X-Correlation-Id |
|---|---|---|---|---|---|
| API Gateway | Validates/forwards | Validates | Validates | Generates if missing | Generates if missing |
| Auth Service | Optional depending on endpoint | Yes with body | Yes | Yes | Yes |
| User Service | Yes on protected endpoints | Yes with body | Yes | Yes | Yes |
| Listing Service | Yes on protected endpoints | Yes with body | Yes | Yes | Yes |
| Search Service | Depends on endpoint design | Usually not in GET | Yes | Yes | Yes |
| Availability Service | Depends on endpoint | Yes when body exists | Yes | Yes | Yes |
| Booking Service | Yes | Yes | Yes | Yes | Yes |
| Payment Service | Yes or trusted internal auth | Yes | Yes | Yes | Yes |
| Notification Service | Internal only | Yes | Yes | Yes | Yes |

## Example Request

```http
POST /v1/bookings HTTP/1.1
Authorization: Bearer <token>
Content-Type: application/json
Accept: application/json
X-Request-Id: 5f8c1d6e-2e18-4ef8-b1af-92e7db3f7c2a
X-Correlation-Id: 1f0b4e2c-7dd0-42f3-bf61-d1ff8b25f3e9
```

## Example Log Entry

```text
timestamp=2026-03-31T18:10:22Z
service=booking-service
requestId=5f8c1d6e-2e18-4ef8-b1af-92e7db3f7c2a
correlationId=1f0b4e2c-7dd0-42f3-bf61-d1ff8b25f3e9
path=/v1/bookings
method=POST
status=201
```

## Smithy Modeling Guidance

The file `model/common/http-headers.smithy` defines reusable header structures:

- `TraceHeaders`
- `JsonHeaders`
- `AuthHeader`

These structures provide a shared contract, but each operation must still explicitly model the relevant `@httpHeader(...)` members in its input according to the endpoint type.

## Approval Criteria Covered

- Defined list of standard headers
- Mandatory usage documented
- Service-by-service usage documented
- Request tracing and correlation rules documented
- Ready to reference from Smithy models and technical design docs
